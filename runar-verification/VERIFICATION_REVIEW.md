# Rúnar Lean 4 Verification — Independent Audit

Audit date: 2026-05-06. Reviewer: Claude Opus 4.7. Build state: `lake build` succeeds (26/26 jobs, warnings only). All artifacts read; no files modified except this review.

---

## 1. Executive summary

The verification builds, has zero `sorry` / `admit`, and contains a real and substantial body of Lean 4 work — a parser, ANF + Stack + Script syntaxes, an ANF evaluator, a Stack VM, a lowering pass, a 9 800-line peephole module with dozens of soundness lemmas for individual rewrite rules, and a passing 49/49 well-formedness + JSON round-trip suite. **The empirical content is real and useful as a regression check on the Lean port of the compiler.**

What the verification is **not**: an end-to-end soundness proof of the production Rúnar compiler. The package's headline pipeline-level "soundness theorems" (`compile_observational_correct`, `compile_observational_correct_bytes`, `lower_observational_correct`, `peephole_observational_correct`, `emit_observational_correct`) are all stated with hypotheses that already imply the conclusion, and proven by `exact hSimulates` / `successAgrees_refl` / one-line transitivity. They are vacuous as soundness statements — supplying their hypotheses is precisely the unsolved problem. `#print axioms` confirms they pull in only Lean's standard kernel axioms because they delegate every load-bearing claim to a per-call hypothesis that the user must produce.

Independently of the vacuous theorems, the **empirical claim** ("Lean compiler emits same bytes as TS reference for 49/49 fixtures") is also weaker than advertised: the per-PR CI gate (`ci.yml` `runar-verification` job) only validates **34** fixtures by actually invoking the Lean compiler. The other 15 (`cryptoAxiomPending` — every `secp256k1`, every NIST P-256/P-384, every SLH-DSA, every BabyBear, every Merkle fixture) are gated against a constant defined as `include_str("../../conformance/tests/<name>/expected-script.hex")` — i.e. the gate compares the file to itself. Real validation of those 15 fixtures requires `RUNAR_VERIFICATION_FULL=1`, which is `workflow_dispatch` only, takes >1 hour, can be truncated by GitHub's 6 h timeout, and does not gate PR merges.

Most importantly, even if the vacuous theorems were discharged, they would say almost nothing about a deployed Rúnar contract because (a) the verified `compile` is a Lean reimplementation, not extracted from any of the seven production compilers, with no mechanism preventing drift; (b) the Stack VM stubs out OP_CHECKSIG (`opaque ... := false`), OP_CHECKMULTISIG (single-pop "pragmatic stub"), OP_CHECKPREIMAGE (`evalValue` mocks to `true` while the Stack VM honours an axiom), OP_CODESEPARATOR (no-op, breaks sighash coverage), OP_BIN2NUM, OP_NUM2BIN, OP_INVERT, OP_AND, OP_OR, OP_XOR (all return constant abstract values); (c) there is no `parseScript` decoder, so the relationship between `compile p : ByteArray` and on-chain execution is never formalised; (d) there is no formal Bitcoin Script semantics — `Stack.Eval.runOps` is an unverified transcription of "the Rúnar-emitted subset"; (e) the type system covers 5 of 18 ANFValue constructors and the documented `type_preservation` stand-in is `: True := trivial`.

**Single most important thing to fix first.** Stop calling `lower_observational_correct`, `compile_observational_correct`, and `compile_observational_correct_bytes` "theorems with explicit hypotheses." They are tautologies. Either rename them (e.g. `lower_observational_correct_skeleton`) and label them as scaffolding, or replace them with an honest statement of what is currently provable: a per-fixture `compile p_i = expected_hex_i` table for the 34 fixtures the regression gate actually runs. The current naming materially overstates the verification's reach to anyone reading `README.md` + `TRUST_MANIFEST.md` + `Pipeline.lean` in isolation.

## 2. Trusted Computing Base

The verification's TCB, in increasing order of project-specific risk:

1. **Lean 4 kernel** (`leanprover/lean4:v4.29.1`, pinned in `lean-toolchain`). Standard, acceptable.
2. **Standard kernel axioms** — `propext`, `Classical.choice`, `Quot.sound`. All `#print axioms` outputs reduce to these. Acceptable.
3. **No external dependencies.** `lake-manifest.json` lists zero packages — no mathlib, no std4, no batteries. Reduces external trust to zero, but means every basic structure (lists, ByteArrays) is from Lean core.
4. **Build tooling.** `lakefile.lean` is small and trivial; `lake build` reproduces from a fresh clone (verified).
5. **`opaque` declarations (9 total).**
   - `Crypto.sha256`, `Crypto.ripemd160`, `Crypto.hash160`, `Crypto.hash256` (`ANF/Eval.lean:231-234`) — defaults to `ByteArray.empty`.
   - `Crypto.checkSig` (`ANF/Eval.lean:302`) — defaults to `false`.
   - `Stack.Eval.checkMultiSigStub` (`Stack/Eval.lean:243`) — defaults to `false`.
   - `Typed.builtinSig` (`ANF/Typed.lean:133`) — `Option FuncSig`, no body. Effectively means `HasType.callT` is never instantiable for any concrete `func`.
6. **`axiom` declarations (57 total, in `ANF/Eval.lean` and one in `Stack/Peephole.lean`).** None are pulled into the top-level theorems' axiom dependency, because the top-level theorems delegate everything via hypotheses (see `compile_observational_correct`'s axiom set below). They *are* pulled into the peephole-soundness theorems via `hash256_eq_double_sha256`. All crypto axioms are **bare function signatures** with no soundness lemmas — no determinism, no group laws, no collision-resistance, no FIPS 205 EUF-CMA, no group axioms, no curve-equation invariants. As stated in `TRUST_MANIFEST.md` §3, "These have no soundness lemmas." Listed in full in §4 of this review.
7. **The Bitcoin Script semantics model** (`Stack.Eval.runOps` and its dispatch table in `Stack/Eval.lean:245-498`). **This is the single largest unverified item.** It is hand-written Lean, never compared to the BSV consensus implementation, and explicitly stubs out 7+ opcodes. See §5 finding S-7.
8. **The relationship between `RunarVerification.Pipeline.compile` and the seven production compilers** (TS, Go, Rust, Python, Zig, Ruby, Java). There is no extraction, no shared source, and no automatic check that the Lean port and any production compiler agree on any binding's lowering. The 34/49 byte-exact regression gate against the *TS* reference is the only correspondence — and the TS reference itself is one of seven implementations, the others entirely unaudited.
9. **Implicit:** the `partial def` evaluators (`evalValue`, `evalBindings`, `runLoop`, JSON `fromJsonANFValue?` / `toJsonANFValue`, plus several peephole sub-passes including `chainFoldFixpointFlat`, `rollPickFixpointFlat`). `partial def`s are not subject to termination checking and cannot drive structural induction; reductions must go through definitional unfolding. Inductive proofs about programs that flow through these definitions are not currently possible.

**Naive read:** "Top-level theorems depend only on Lean's standard axioms; the project-specific `hash256_eq_double_sha256` axiom only enters peephole-internal soundness theorems." This sounds excellent but is misleading: the top-level theorems delegate every project-specific assumption into a hypothesis, so the kernel never sees the assumption. See finding C-1.

The 57 crypto axioms in `ANF/Eval.lean` (sha256Compress, blake3Hash, ecAdd, ecMul, p256Add, p256Mul, p384Add, p384Mul, verifyECDSA_P256, verifyECDSA_P384, verifyWOTS, verifySLHDSA_SHA2_{128,192,256}{s,f}, merkleRootSha256, merkleRootHash256, verifyRabinSig, extractVersion, extractHashPrevouts, extractHashSequence, extractOutpoint, extractInputIndex, extractScriptCode, extractAmount, extractSequence, extractOutputHash, extractLocktime, extractSigHashType, checkMultiSig, checkPreimage, buildChangeOutput, computeStateOutput, bbFieldAdd, bbFieldSub, bbFieldMul, bbFieldInv, ecMulGen, ecNegate, ecOnCurve, ecModReduce, ecEncodeCompressed, ecMakePoint, ecPointX, ecPointY, p256MulGen, p256OnCurve, p256EncodeCompressed, p384MulGen, p384OnCurve, p384EncodeCompressed) **do not appear in `#print axioms compile_observational_correct`**. They are reachable through `Stack.Eval.runOpcode "OP_*"` only when an evaluation actually reaches them, and the top-level theorems never force evaluation — they push the success-bit equivalence into a hypothesis.

## 5. Specification adequacy assessment

**Question: does the top-level theorem actually rule out the bug classes Rúnar cares about?** Working through them:

* **Mis-compilation (Lean compiler emits wrong bytes for some valid contract).** The empirical 34/49 byte-exact gate catches this for fixtures in the gated set. **Does not catch** mis-compilation in any of the 15 cryptoAxiomPending fixtures (default CI gates a tautology), and does not catch any Lean–vs–TS divergence in fixtures the conformance suite doesn't have. The vacuous theorems do not contribute. Verdict: partial empirical coverage (34/49), zero theorem coverage.
* **Stack underflow at runtime.** The Stack VM detects underflows and returns `EvalError.unsupported`. There is no theorem of the form "if `WF.ANF p`, then `runOps (lower p)` never returns `unsupported "stack underflow"`." Verdict: not addressed.
* **Wrong sighash.** No sighash model. `OP_CODESEPARATOR` no-op'd; preimage stubbed empty; 11 BIP-143 extractors are bare `axiom`s with no consistency lemmas (e.g. no `extractAmount (Crypto.computeStateOutput sk amount idx) = amount`). Verdict: not addressed at all.
* **Broken EC operations.** `ecAdd`, `ecMul`, `ecMulGen`, `ecNegate` are bare axioms with no group laws. The lowering pass `Stack/Ec.lean` ports the codegen, but the resulting Stack-IR's behaviour against `Crypto.ecAdd` axiom is not constrained. Verdict: codegen mirrors TS reference; mathematical correctness unverified.
* **Lost funds (i.e. spendable when shouldn't be, or vice versa).** Requires (a) sighash correctness, (b) `OP_CHECKSIG` modeling DL-soundness, (c) state-continuation axiom faithfulness, (d) faithful BSV consensus model. None of these is present. Verdict: not addressed.
* **Cross-compiler divergence (TS, Go, Rust, Python, Zig, Ruby, Java disagree on the same `.runar.*` source).** This is Rúnar's project-level safety claim ("Stack-IR + hex parity scoped to non-allowlisted fixtures, all 7 tiers"). The Lean verification compares `Lean ↔ TS` only. The other 21 cross-compiler edges (Go↔Rust, Go↔Python, etc.) are validated by the separate `conformance/runner/runner.ts` suite, not by `runar-verification/`. Verdict: not addressed by this verification at all.
* **Peephole rule introduces a behaviour change.** The 25 individual `_pass_sound` peephole theorems (in `Stack/Peephole.lean`) **are real** and prove that each rewrite preserves observational equivalence on `wellTypedRun` inputs, modulo the `hash256_eq_double_sha256` axiom for the double-SHA-256 fusion. This is the strongest content of the verification. Verdict: addressed for individual rules; the composition theorem `peepholePassFullPlus_sound` covers 12 of 19 rules; the remaining 7 rules' soundness is tracked in the `Stage A/B/C/D` substrate but not yet composed.

**Adversarial check — "how could the top-level theorem be true and Rúnar still be broken?"** Many ways:

1. The user supplies a deliberately wrong `hSimulates` (the theorems' hypotheses can be discharged by `Iff.refl` for any program where evaluation diverges identically on both sides, including programs that fail for unrelated reasons).
2. The Lean `compile` matches the TS reference byte-exactly but differs from the Go / Rust / Python / Zig / Ruby / Java implementations (six of seven production tiers unaudited).
3. The 15 cryptoAxiomPending fixtures' Lean compileHex output has silently changed since the last `RUNAR_VERIFICATION_FULL=1` regen run, and the default CI gate (which compares `expected-script.hex` to itself) misses it.
4. The Lean Stack VM handles `OP_CHECKSIG` as `false`, the TS reference compiler emits a contract whose locking script depends on a real signature; the Lean port matches byte-for-byte, the contract works on testnet, but the verification's "successAgrees" predicate is satisfied because both sides return `false` for the test inputs in the verification.
5. A future Rúnar feature requires a new opcode (e.g. `OP_CHECKDATASIGVERIFY`) — the Stack VM falls through to `EvalError.unsupported`, the Lean compiler emits the opcode in `expected-script.hex`, the byte-exact gate goes green for the new fixture.

Each of these is consistent with all the green checks in `lake build` and `pipelineGolden`.

## 7. Bottom line

This package is a real and substantial body of Lean 4 code that produces:

* A working ANF / Stack / Script syntax tree, JSON parser, and well-formedness predicate (`49/49` empirical).
* A Lean reimplementation of the lowering + peephole + emit pipeline that matches the TS reference byte-for-byte on **34** fixtures gated in CI and **claims** to match all 49 (the last 15 are gated tautologically; the real check exists only in a workflow_dispatch job).
* A genuine and non-trivial set of soundness lemmas for individual peephole rewrite rules (~25 `_pass_sound` theorems in `Stack/Peephole.lean`, one composition theorem `peepholePassFullPlus_sound` covering 12 of 19 rules, depending only on `hash256_eq_double_sha256`).
* Stage A/B/C/D scaffolding in `Stack/Agrees.lean` toward a real lowering simulation theorem.

It does **not** produce, despite the documentation framing:

* An end-to-end soundness theorem connecting ANF programs to deployed Bitcoin Script bytes. The four pipeline-level theorems are tautologies; one is misnamed.
* A meaningful TCB inventory at the top-level theorem boundary (the `#print axioms` output is artificially clean because all project-specific assumptions are pushed into hypotheses).
* Any cross-compiler guarantee. Six of seven production compilers are not in the verification's scope.
* A faithful Bitcoin Script semantics. Several opcodes are stubbed; sighash is unmodelled; `OP_CODESEPARATOR` is a no-op.
* A barrier against drift between the verified Lean compiler and the production compilers. There is no extraction.

The most useful short-term action is documentation honesty: replace the "Phase 6 closeout — capstone axiom replaced with theorem-with-hypothesis" framing with an accurate "the soundness theorem is a tautology pending discharge of `hSimulates`," fix the README/TRUST_MANIFEST/PipelineGolden disagreement on the byte-exact count, and either remove or downgrade `cryptoAxiomPendingExpected`'s tautological comparison so the `49/49` number reflects work CI actually does. The most useful medium-term action is to land the Stage B/C/D discharge for the basic-`SimpleANF` subset (the substrate is in place in `Stack/Agrees.lean`); that converts `lower_observational_correct` from a tautology into an actual theorem on a useful program class.
